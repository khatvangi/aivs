# AIVS ↔ PROV-AGENT alignment

**Purpose.** Position AIVS technically and strategically relative to PROV-AGENT
(Souza et al., IEEE eScience 2025, arXiv:2508.02866), the only directly
adjacent prior art for tracking AI involvement in scientific workflows. This
document is internal-facing: it informs schema decisions and provides
positioning text reusable in the README, future papers, and grant applications.

## Summary

PROV-AGENT and AIVS solve adjacent but distinct problems. PROV-AGENT captures
agentic workflow execution in real time during runs at HPC facilities. AIVS
audits AI-mediated research workflows after the fact, anchored in scientific
manuscripts. The two are complementary; AIVS should explicitly build on
PROV-AGENT-compatible foundations rather than reinvent them.

The clean framing: **PROV-AGENT is the workflow-capture layer. AIVS is the
manuscript-anchored audit layer that sits above it.** Either can be used
without the other, but they compose naturally.

## Problem-space differences

|                              | PROV-AGENT                      | AIVS                                    |
| ---------------------------- | ------------------------------- | --------------------------------------- |
| **When captured**            | Real-time, during execution     | Retrospective, post-hoc                 |
| **What is anchored to**      | Workflow execution graph        | Manuscript claims                       |
| **Primary audience**         | HPC workflow developers         | Researchers, journal editors, reviewers |
| **Primary question**         | What did the agents do?         | How does this paper's claim chain back to AI involvement, and where? |
| **Deployment locus**         | Compute facility infrastructure | Author-side tool against own work       |
| **Capture method**           | Instrumentation (decorators, LLM wrappers) | Adapter ingestion of existing logs |
| **Vocabulary**               | Fixed PROV-AGENT schema         | Open vocabulary, evolves via SchemaDelta |
| **Fidelity handling**        | Assumes instrumentation present | Explicit CaptureTier 0–3 gradient       |
| **Verification status**      | Not modeled                     | First-class field on Decision           |

## PROV-AGENT's model (summary for reference)

PROV-AGENT extends W3C PROV's three core classes (Entity, Activity, Agent) as
follows:

**New Activity subclasses**
- `Campaign`, `Workflow`, `Task` — standard workflow structures
- `AgentTool` — one tool execution by an AI agent
- `AIModelInvocation` — one call to a foundation model

**New Entity / DataObject subclasses**
- `DomainData`, `SchedulingData`, `TelemetryData`
- `Prompt`, `ResponseData`
- `AIModel` — model metadata (provider, name, parameters, temperature)

**New Agent subclasses**
- `Person`, `Organization`, `AIAgent`

**Key relationships (standard PROV constructs)**
- `AIAgent` `wasAssociatedWith` `AgentTool`
- `AgentTool` `wasInformedBy` `AIModelInvocation`
- `AIModelInvocation` `used` `Prompt`, `used` `AIModel`, `wasGeneratedBy` `ResponseData`
- `ResponseData` `wasAttributedTo` `AIAgent`
- Data outputs generated by agent tools `wasAttributedTo` `AIAgent`

**Implementation:** Flowcept (`github.com/ORNL/flowcept`), federated
broker-based provenance framework. MCP-aware. Decorator-based instrumentation
(`@flowcept_agent_tool`). LLM wrapper (`FlowceptLLM`).

## Entity-level mapping (AIVS → PROV-AGENT)

| AIVS entity / field                       | PROV-AGENT counterpart                                | Notes |
| ----------------------------------------- | ----------------------------------------------------- | ----- |
| `Event` (action=prompt)                   | `Prompt` (Entity) generated by `AIModelInvocation`   | Adapter must reconstruct the invocation grouping |
| `Event` (action=generate)                 | `ResponseData` (Entity) generated by `AIModelInvocation` | Same |
| `Event` (action=execute, AI agent)        | `AgentTool` execution                                 | Direct |
| `Event` (action=run, non-AI)              | `Task`                                                | Direct |
| `Event` (action=edit, human)              | `Activity` `wasAssociatedWith` `Person`               | Generic PROV Activity, no PROV-AGENT subclass |
| `Actor.actor_type=HUMAN`                  | `Person` Agent                                        | Direct |
| `Actor.actor_type=AI_GENERATIVE`          | `AIAgent` with `AIModelInvocation`                    | AIVS distinguishes advisory/generative/autonomous; PROV-AGENT collapses these into AIAgent + context. Mapping is one-to-many. |
| `Actor.actor_type=AI_ADVISORY`            | `AIAgent` whose `ResponseData` informs a human-authored activity | Lossy: PROV-AGENT does not mark "advisory" as such; the advisory relation is implicit in the graph topology. |
| `Actor.actor_type=AI_AUTONOMOUS`          | `AIAgent` whose `AgentTool` executions are not human-reviewed | Same; lossy |
| `Actor.actor_type=SYSTEM`                 | `Person`/`Organization` Agent attached to automated tooling | Approximate |
| `Actor.model_metadata`                    | `AIModel` Entity                                      | AIVS has unstructured dict; PROV-AGENT has typed fields. AIVS v0.2 candidate refinement. |
| `Evidence`                                | (composite: cluster of related Entities + Activities) | No first-class equivalent; AIVS aggregates events; PROV-AGENT users query relationships directly. |
| `Decision`                                | (no direct equivalent)                                | PROV-AGENT treats agent decisions as `DomainData` `wasAttributedTo` `AIAgent`, without classification, verification status, or schema-gap marking. AIVS extends here. |
| `Decision.decision_type`                  | (no equivalent)                                       | AIVS open vocabulary; PROV-AGENT has none |
| `Decision.verification_status`            | (no equivalent)                                       | AIVS extension |
| `Decision.schema_gap`                     | (no equivalent)                                       | AIVS extension |
| `Claim`                                   | (no equivalent)                                       | Manuscript scope is outside PROV-AGENT |
| `AuditArtifact`                           | (no equivalent)                                       | PROV-AGENT produces a queryable provenance graph; AIVS produces a packaged retrospective audit |
| `SchemaDelta`                             | (no equivalent)                                       | PROV-AGENT schema is fixed |
| `CaptureTier`                             | (no equivalent)                                       | PROV-AGENT assumes full instrumentation |

## Gaps PROV-AGENT does not address (where AIVS adds value)

1. **Manuscript anchoring.** PROV-AGENT's graph terminates at the workflow.
   It has no concept of a manuscript claim or a section/paragraph anchor. AIVS
   `Claim` with `location` and `upstream_decision_ids` is the missing layer
   between "what the agents did" and "what the paper says."

2. **Decision as a first-class typed entity.** PROV-AGENT records that an
   agent produced an output that was used downstream. It does not classify
   that output as a methodological decision, does not record verification
   status, and does not flag novel patterns the schema doesn't cover.

3. **Retrospective extraction.** PROV-AGENT requires instrumentation at run
   time. If the user did not decorate their code with `@flowcept_agent_tool`,
   no provenance exists. AIVS's adapter model ingests whatever logs *do*
   exist (git, chat sessions, notebooks) and is honest about what it could
   not recover.

4. **Capture-fidelity gradient.** PROV-AGENT does not represent partial
   capture. A workflow either was instrumented or it wasn't. AIVS's tier 0–3
   makes the fidelity of each decision and claim explicit in the audit
   output.

5. **Open vocabulary, schema deltas.** PROV-AGENT's schema is closed and
   designed up-front. AIVS's vocabulary accretes from cases via SchemaDelta
   and is descriptive, not prescriptive.

6. **Finer-grained actor taxonomy.** PROV-AGENT has a single `AIAgent` class;
   advisory vs. generative vs. autonomous is implicit. AIVS's
   `ActorType` enum makes this distinction explicit, which matters for the
   audit question "did a human review this AI output before it became a
   decision?"

## Refinements PROV-AGENT suggests for AIVS (candidates for v0.2)

The meta-schema shipped in v0.1.0 is workable but loses information that
PROV-AGENT models cleanly. The following are candidate refinements; none
require breaking changes if introduced as optional fields.

1. **Promote AIModel to a first-class entity.** Replace
   `Actor.model_metadata: dict[str, Any]` with a typed `AIModel` class
   carrying `provider`, `name`, `version`, `parameters` (with `temperature`,
   `top_p`, etc. as known keys). Backwards-compatible: keep `model_metadata`
   as a fallback when an adapter cannot produce a structured `AIModel`.

2. **Add an AIModelInvocation activity type.** A grouping that ties together
   a `Prompt` event, a `ResponseData` event, and an `AIModel`. Currently
   AIVS represents prompt and response as separate events linked only by
   proximity in the source stream; the explicit grouping makes downstream
   queries cleaner and mirrors PROV-AGENT.

3. **Add an AgentTool activity type.** For agentic settings where an AI
   agent invokes tools (Claude Code's bash, file_edit, search calls; CrewAI
   tool calls; LangChain tools). One `AgentTool` execution composes one or
   more `AIModelInvocation` events and any tool I/O events.

4. **Adopt PROV relationship names as the canonical vocabulary for relations
   between Events / Evidence / Decisions.** Use `used`, `wasGeneratedBy`,
   `wasAttributedTo`, `wasAssociatedWith`, `wasInformedBy`. AIVS already
   plans PROV-O export; adopting the names internally reduces export-time
   translation.

5. **Cite the FAIR4ML metadata schema and PROV-ML in the export layer.**
   These are complementary to PROV-AGENT for ML-specific artifacts (model
   training metadata, dataset cards). The RO-Crate exporter should
   accommodate them as profile extensions.

These refinements are not blocking. The v0.1.0 schema is sufficient for the
first retrospective audit. v0.2 can consolidate the alignment.

## Interoperability options

Three plausible postures, in increasing order of integration:

**A. Independent, schema-aligned.** AIVS uses PROV-AGENT-compatible
terminology and entity names; export to PROV-O / RO-Crate maps cleanly. No
runtime dependency on Flowcept. (Recommended for v0.2.)

**B. Flowcept as an evidence adapter.** A `flowcept_adapter` that reads a
Flowcept provenance database and emits AIVS Events. This makes AIVS the
post-hoc audit layer over a PROV-AGENT-instrumented workflow. Useful for
HPC researchers already using Flowcept; not necessary for the typical
author-side retrospective use case.

**C. Joint authorship / shared schema.** Coordinate with the ORNL group on
a unified extension to W3C PROV that covers both real-time capture
(PROV-AGENT) and retrospective manuscript-anchored audit (AIVS). High
political and coordination cost; high standards-legitimacy payoff.
Premature now; a candidate after AIVS has a first working audit to point
to.

Recommended path: **A first, B as needed, C only after empirical
demonstration of AIVS value.** Do not pursue C before having something
PROV-AGENT's authors would want to coordinate with.

## Strategic positioning text (reusable)

Short form (one sentence, for README, abstracts, grant intros):

> AIVS is a retrospective, manuscript-anchored audit framework for
> AI-mediated research workflows. It complements real-time agentic
> provenance systems such as ORNL's PROV-AGENT (Souza et al., 2025) by
> providing the missing layer between in-workflow capture and the
> verifiability of published scientific claims.

Medium form (paragraph, for project landing pages, paper introductions):

> The integration of AI tools into scientific research has produced a
> mismatch between disclosure policies (which require attestation but
> cannot verify) and runtime provenance systems (which capture rich data
> but require instrumentation absent in most existing research). AIVS
> occupies this middle layer. It is descriptive rather than prescriptive,
> retrospective rather than real-time, and anchored to scientific
> manuscripts rather than to workflow execution graphs. Where ORNL's
> PROV-AGENT [Souza et al., IEEE eScience 2025] captures AI agent
> interactions during cross-facility execution, AIVS produces audit
> artifacts after the fact, ingesting whatever logs exist (chat
> transcripts, git history, notebooks, environment metadata) and reporting
> honestly on the capture fidelity achieved. The two systems are
> complementary; AIVS extends PROV-AGENT-compatible terminology and can
> optionally consume Flowcept-generated provenance as evidence.

Long form (for a paper's Related Work section): adapt the "Problem-space
differences" table and the "Entity-level mapping" section above.

## Open questions to revisit after the first AIVS retrospective audit

1. Does the AIVS retrospective extractor naturally produce
   PROV-AGENT-compatible structures, or are there real impedance
   mismatches we haven't anticipated?
2. Is `AgentTool` the right granularity for capturing Claude Code's tool
   calls, or do we need a finer-grained primitive for streaming /
   multi-step tool sessions?
3. Should AIVS adopt MCP terminology (`tool`, `resource`, `prompt`,
   `context`) directly, given MCP is becoming standard?
4. Is option B (Flowcept-as-adapter) interesting enough to pursue once the
   meta-schema stabilizes, or is option A sufficient indefinitely?
